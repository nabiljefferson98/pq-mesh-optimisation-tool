# Data Guide

`data/`

This folder contains all project data: input meshes used to test and benchmark
the optimiser, and output files produced by the optimisation pipeline. It is
divided into two subfolders: `input/` for source meshes and `output/` for
all generated results.

---

## Directory Structure

```
data/
├── DATA_GUIDE.md                     <- this file
├── input/
│   ├── INPUT_MESHES.md               <- catalogue of all input meshes
│   ├── generated/                    <- synthetic meshes produced by generation scripts
│   └── reference_datasets/           <- external meshes from public geometry repositories
└── output/
    ├── optimised_meshes/             <- OBJ exports from the optimiser
    ├── panels/                       <- DXF and SVG fabrication-ready flat panel exports
    ├── benchmarks/                   <- timing data, convergence plots, benchmark tables
    ├── weight_sensitivity/           <- weight sensitivity study data and Pareto plots
    └── tables/                       <- LaTeX and CSV summary tables for the dissertation
```

---

## Input Subfolders

### `input/generated/`

Contains synthetic quad meshes produced by the mesh generation scripts in
`scripts/mesh_generation/`. These meshes cover a range of surface types
(planar grids, saddle surfaces, cylindrical patches, torus patches, Scherk
minimal surfaces, and spherical caps) at various resolutions and noise levels.
They are the primary test meshes used in the dissertation benchmarks.

To regenerate all meshes in this folder:

```bash
python scripts/mesh_generation/generate_test_meshes.py
```

See `data/input/INPUT_MESHES.md` for the full catalogue with vertex counts,
face counts, surface types, and noise levels.

### `input/reference_datasets/`

Contains quad meshes sourced from established public 3D geometry repositories,
including Keenan Crane's geometry processing library and the Stanford 3D
Scanning Repository. These meshes represent real-world or community-standard
models and are used to evaluate the pipeline beyond synthetic test cases.

See `data/input/INPUT_MESHES.md` for the full catalogue with source
attributions and notes on each mesh.

---

## Output Subfolders

### `output/optimised_meshes/`

Contains OBJ files exported after optimisation and, where generated, PNG
convergence plots showing how the total energy decreased across iterations.
Filenames follow the pattern `<input_name>_optimised.obj`.

### `output/panels/`

Contains fabrication-ready flat panel layouts exported by `panel_exporter.py`.
Each optimised quad face is unfolded into a flat quadrilateral using SVD-based
orthogonal projection and written to either a DXF file (for CAD software) or
an SVG file (for visual inspection). Files are named after the source mesh
with a `_panels` suffix.

### `output/benchmarks/`

Contains performance measurement data produced by the benchmarking scripts in
`scripts/benchmarking/`. This includes:

- `performance_data.json`: raw timing results for each mesh and backend.
- Convergence PNG plots: energy versus iteration for each benchmarked mesh.

### `output/weight_sensitivity/`

Contains the results of the weight sensitivity analysis, including:

- `weight_sensitivity_data.json`: energy values across the weight parameter
  space.
- `plots/`: Pareto front plots showing the trade-off between planarity and
  closeness.
- A human-readable summary report in `.txt` format.

### `output/tables/`

Contains tabular summaries of benchmark results in two formats:

- `benchmark_table.tex`: a LaTeX-formatted table for direct inclusion in the
  dissertation.
- `benchmark_table.csv`: the same data in CSV format for spreadsheet review.

---

## Version Control Policy

Large binary files (OBJ meshes, PNG images, DXF files, SVG files) under
`output/` are excluded from version control via `.gitignore`. Only structured
data files (`.json`, `.csv`, `.tex`, `.txt`) are committed. This keeps the
repository size manageable while preserving reproducibility: all output files
can be regenerated from the committed scripts and input meshes.

The `input/generated/` folder is also excluded from version control because
its contents can be fully regenerated by running the generation script. The
`input/reference_datasets/` folder contains meshes that must be downloaded
from their respective sources; see `INPUT_MESHES.md` for attribution and
download instructions.

## Output File Format Summary

| Format | Subfolder | Description |
|---|---|---|
| `.obj` | `optimised_meshes/` | Optimised mesh geometry with quad topology preserved |
| `.dxf` | `panels/` | Flat panel layout for CAD software (AutoCAD R2010 format) |
| `.svg` | `panels/` | Visual preview of flat panel layout |
| `.json` | `benchmarks/`, `weight_sensitivity/` | Raw performance and sensitivity data |
| `.png` | `benchmarks/`, `weight_sensitivity/plots/` | Convergence and Pareto plots |
| `.tex` | `tables/` | LaTeX-formatted dissertation table |
| `.csv` | `tables/` | Spreadsheet-friendly version of the benchmark table |
| `.txt` | `weight_sensitivity/` | Human-readable weight sensitivity report |

## Adding New Input Meshes

When adding a new mesh to either input subfolder:

1. Save it to `data/input/generated/` if produced by a generation script, or
   `data/input/reference_datasets/` if sourced externally.
2. Use the naming convention `<surface_type>_<resolution>_<variant>.obj`,
   for example `saddle_10x10_noisy.obj` or `cone_8x8_clean.obj`.
3. Ensure the mesh is in Wavefront OBJ format with exactly four vertex indices
   per face. See `INPUT_MESHES.md` for the required format specification.
4. Update the mesh catalogue table in `INPUT_MESHES.md`.
5. If the mesh is intended for regression testing, add a corresponding entry
   in the relevant test file under `tests/`.
